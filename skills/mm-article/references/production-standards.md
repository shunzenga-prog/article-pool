# mm-article 生产级标准

本文件是 `mm-article` 的生产级质量门禁。执行真实文章创作、审阅、配图、发布时必须读取；它不是风格建议，也不是可选 checklist。

## 生产级总则

- 先确定平台，再确定格式；公众号、CSDN、小红书是三套不同产物，禁止机械互转。
- 先验证事实，再写观点；没有来源的事实不能进入正文。
- 先形成研究问题，再组织材料；深度文章不能只是来源摘要加普通评论。
- 先确认正向价值，再说明边界；新闻发布、模型发布、产品发布类文章默认从客观事实、积极意义和用户/消费者能解决什么问题出发，深度不等于抬杠。
- 先规划视觉，再生成或截图；图片不是最后补洞。
- 先过审阅，再发布；文章写完不等于任务完成。
- 旧流程继续可用；`mm-article` 只能新增语义层和读取旧工具，不能改变旧 skill 或旧脚本默认行为。

## P0 阻断门禁

以下任一项失败，必须停止流程，不允许发布：

- 平台未确定，或输出格式与平台不一致。
- 关键事实没有可靠来源，或来源时间无法确认。
- 新闻/产品/模型/价格/政策等时效性信息未重新搜索。
- 模型发布、产品发布、技术趋势等深度文章没有研究问题、指标拆解、机制解释、限制边界、正向价值和读者决策清单。
- 标题不满足平台约束，或含弱词、错误品牌名、AI 味句式。
- 公众号 HTML 违反微信 CSS 硬规则，或正文首屏重复系统标题。
- 生成图被当作截图、新闻图、官方图或现实证据使用。
- 图片来源门禁未通过，或在评分前发现来源为 `geometric`、`fallback_pattern`、`fallback_auto`、`legacy_without_reason`、缺少来源记录。
- 原文证据截图缺少来源记录、裁剪范围泄露登录态/私信/侧栏噪声，或截图不能支撑正文 claim。
- `visual_slot_conflict`：同一段落、同一视觉槽位出现相邻图片或连续图片堆叠，且没有独立叙事作用。
- 封面没有从文章主张提炼 brief，或封面泛化成普通素材站科技图。
- 内置生图工具未暴露模型名时，在报告、正文或回复里声称具体底层模型名。
- 正文、图注、alt、卡片或结尾出现工作流提示词、制作者备注、审阅说明或图片处理说明。
- 正文、截图或日志暴露 API Key、Cookie、token、账号、隐私、本地绝对路径。
- 发布前审阅未通过，或发布命令没有返回草稿 ID / 文章 ID。

## 证据与时效

### 来源优先级

1. 官方和一手来源：官网、官方博客、release note、论文、GitHub 仓库、监管公告、公司公告。
2. 权威二手来源：主流科技媒体、行业报告、学术数据库、可信新闻源。
3. 社区信号：Hacker News、Reddit、知乎、微博、X、论坛。只能辅助判断热度、争议和用户反馈，不能单独支撑事实。

### 时效分类

- 24 小时内：实时热点，可以按新闻写。
- 24-48 小时：近期热点，需要说明进展或影响。
- 超过 48 小时：避免伪装成新闻，转为分析、复盘、教程或方法论。

### 证据台账

每个关键事实都要能回到证据台账：

```text
事实：
来源：
发布时间：
检索时间：
支持范围：
限制或争议：
是否可用于正文：
```

### 原文证据截图

当正文需要引用原始网页、X 发帖、论坛发言、官方公告或论文页面时，必须在起草前形成原文证据截图记录：

- 截图记录写入 `source_capture.json`，并进入 `source_capture_artifacts`。
- 每条记录必须包含来源 URL、截图时间、是否翻译、裁剪范围、支持的 claim、读者可见用途和 `visual_slot_id`。
- 社交平台截图只保留头像、账号、正文和必要时间信息；不要把浏览器导航、侧栏、视频大块区域、推荐流、登录态、通知或私信截入正文素材。
- 原文证据截图只能证明“来源页面这样写过”，不能替代官方公告、论文或最新时效验证。
- 图注不得出现“裁切版”“视觉计划”“图片请求”等生产备注。

### 本地知识检索

- `local_kb_retrieval`：本地知识库、历史文章和内部资料必须通过 `article-research-kb` 渐进检索，并记录路径、位置、支持的 claim 和限制。
- 本地资料只能证明“已有材料如何描述”，不能替代官方来源、最新公告和时效性验证。
- 如果本地证据缺路径、缺位置或与外部事实冲突，不能直接进入正文；先补证据或降级为背景线索。

## 内容与标题

### 标题

- 公众号标题必须 20-30 字；不足 20 字直接重写。
- 至少 2 个钩子：数字、反常识、紧迫感、承诺价值、制造好奇、身份认同、结果对比、具体对象。
- 禁用弱词：浅谈、浅析、关于、漫谈、初探、之我见。
- 禁止 AI 味标题：`从 X 看 Y 的 Z`、`X 的机遇与挑战`、`关于 X 的若干思考`。
- 品牌、产品、模型、人物和机构名称必须准确。

### 内容提示词

起草前必须生成内容提示词，至少包含：

- 读者画像和阅读收益。
- 研究问题、文章角度和不写什么。
- 证据清单和事实边界。
- 已有原文证据截图及其 `source_capture_artifacts`、支持 claim 和拟使用视觉槽位。
- 指标定义、机制解释、正向价值、限制边界和读者决策。
- 开头策略、段落推进、转场计划和生硬度自检。开头策略必须先判断文章是事实驱动还是体验驱动；新闻发布、模型发布、政策公告等文章可以直接采用事实导语，不得为了制造钩子硬编读者体验。
- 写作倾向：以客观事实和正面意义为主，说明它能给消费者/用户带来什么便利、效率、体验或选择空间；限制边界只能中性说明，不能喧宾夺主。
- 结构节奏和关键转折。
- 重点句突出计划：每个大节最多 1 句、全文通常不超过 5-7 处；优先突出关键数据、机制转折、适用边界和行动建议，不突出空泛形容词或整段文字。
- 视觉计划和平台格式。

### 深度研究

热点和发布类深度稿不能满足于“来源可靠 + 口语化”。进入起草前必须建立研究框架：

- 明确 `root_question`：读者真正想知道什么。
- 拒绝浅层角度：至少说明为什么不写“发布了什么”“又快又强”“国产模型崛起”等泛化角度。
- 拆解关键指标：说明指标衡量什么、不衡量什么，以及不能推出什么。
- 解释机制链路：把事实背后的工程、产品、商业或政策链路写清楚。
- 放入参照系：同类产品、历史版本、行业趋势、论文/工程实践至少选一种。
- 说明限制边界：限制、反例、未验证点和落地风险只能作为中性边界，不得写成抬杠、唱反调或莫名反思。
- 给出读者抓手：判断清单、迁移建议、验证方法或行动边界。

如果正文只复述来源信息，或每节只把事实换一种说法，判定为深度不足，必须回到研究阶段。
如果正文为了“深度”强行否定发布价值、压低积极意义或把正常边界写成泼冷水，也判定为写作策略失败，必须重写。

### 写作手感

深度文章不是把研究提纲排成正文。起草前必须建立写作策略：

- 开头要选择合适导语：事实驱动文章先交代公开事实和来源，体验驱动文章再进入读者熟悉的具体场景。
- 不允许一上来替读者总结感受、宣布观点或解释“应该怎么看”。
- 发布类文章要先建立正向价值路径：事实是什么、它可能改善什么体验、消费者/用户能少等什么或少做什么；再补充适用范围。
- 必须写出 `nut_graf`：让读者知道这篇文章的核心问题，但不能像报告摘要。
- 每一节必须有推进动作：提出冲突、解释原因、补一个限制、换到应用场景、给出判断。
- 转场句要像自然说话，不要反复使用“这意味着”“值得注意的是”“由此可见”“总的来说”。
- 发布、测评、模型对比类文章不要用第一人称观点框架收束，例如“我的判断”“我的观点”“我认为”“我对这次发布的整体评价”。结论应由证据、使用场景和读者决策自然推出，避免把事实分析写成作者个人表态。
- 结尾回到读者问题或具体判断，不写宏大口号。

如果文章开头、转场或结尾让人感觉生硬、牵强、刻意引导，即使事实和格式通过，也判定为未通过。

### 叙事结构

- 开头必须有明确钩子，不要从背景套话开始。
- 主体每 300-500 字应有小节推进或视觉/案例支撑。
- 重要观点要包含判断、证据、解释和对读者的意义。
- 对新闻发布、模型发布、产品发布，重点判断应优先回答“它解决了什么用户问题、带来什么积极意义”，不要把正常边界写成反对意见。
- 对发布、测评、模型对比类文章，章节标题和结尾不得用“我的判断”“我的观点”“我认为”“我对这次发布的整体评价”等第一人称观点包装；改用事实归纳、使用场景和读者可执行判断。
- 结尾应自然收束或给出行动，不要空泛总结。
- 读者可见内容不得暴露生产过程。禁止把“事实型图片优先使用官方截图”“不用生成图替代”“裁切版”“视觉计划”“图片请求”“审阅通过”等工作流语言写进正文、图注、alt 或卡片。

## 排版与格式

### 输出目录

- 文章、封面、配图、截图和运行报告必须按 `references/output-contract.md` 落盘。
- 不允许把正式产物散落到仓库根目录、临时目录或 `/Users/mulin/workspace/公众号/文章` 之外的目录。
- 公众号默认参考老文章池命名，但根目录固定为：`/Users/mulin/workspace/公众号/文章/{YYYY}年{MM}月/{MMDD}-{safe_title}.html`。
- 原始 HTML、插图版 HTML、CDN HTML、封面图必须使用同一 basename，便于追踪和发布。
- 正文图片、截图、封面和封面背景都放在文章同一个月份目录下，图片文件平铺命名，不创建正式文章图片子目录。
- 原文证据截图使用 `{MMDD}-{safe_title}-source-{platform}-{NN}-compact.png`，并在 `source_capture.json` 中记录。
- 重点句突出必须使用微信兼容元素：优先 `strong + span` 内联样式或局部 `table` 信息卡；不要在 `<p>` 上设置字体、颜色、背景，不使用整段高亮、渐变、阴影或装饰性强的样式。

### 公众号

- 输出 `.html`。
- 正文第一块可见内容直接进入导语，不重复公众号系统标题。
- 禁止 `<style>` 块，样式必须 inline。
- 禁止用 `<div>` / `<section>` 做容器。
- 文字样式放在 `<span style="...">` 上。
- `<table>` 只做局部卡片，不包裹全文。
- 章节标题必须有 `border-bottom` 下划线。
- 重点句突出要克制：每个大节最多 1 句，全文通常不超过 5-7 处；背景色用浅色底，文字仍保持可读，不新增过多色系。
- 主色系统一，点缀色最多 1 个。
- 发布前运行 `python scripts/review_html.py <文章.html> --json`；教程加 `--tutorial`。
- 宣布交付前运行 `python scripts/validate_mm_delivery.py <文章.html> --run-dir reports/mm-article/<run_id> --title "<title>" --write-report`，确认封面、插图、图片质量、HTML 审阅、流程泄漏、运行报告和发布状态。

### CSDN

- 输出 `.md`。
- 只保留一个 `#` 主标题。
- 用 `##` / `###` 表达层级。
- 代码必须用围栏，表格必须是 Markdown 表格。
- 图片必须是可访问 URL，不写本地路径。

### 小红书

- 标题、正文、封面建议、标签分开。
- 封面标题短、强利益点、移动端可读。
- 不套用公众号 HTML 或 CSDN Markdown。

## 多模态资产

- 事实型视觉优先真实捕获：官方页面、产品 UI、终端输出、代码运行结果、论文图表。
- 概念型视觉可以用生图，但必须标记为概念图。
- 每张图必须有用途、来源、路径、插入位置、事实型/概念型标记。
- 视觉审阅必须先做图片来源门禁，再做视觉得分。顺序固定为 `classify_visual_need → verify_source_provenance → score_visual_quality`。
- 权威人士社交平台发帖、官方公告、论文页和产品页面属于事实证据，优先使用 `source_capture_artifacts` 或真实浏览器截图；概念图不能替代。
- `geometric`、`fallback_pattern`、`fallback_auto`、`legacy_without_reason` 和缺少来源记录的图片在评分前直接阻断，不允许靠尺寸、亮度或构图分数过关。
- 每张图必须有唯一视觉槽位；原文证据截图、教程截图、产品截图和概念插图不得挤在同一视觉槽位。
- 配图必须先产出 `image_requests.json`，且每条请求必须包含 `paragraph_context`；再由 Agent 按所在段落内容生成本地图并写入 `generated_images.json`，最后用 `--use-local-images` 嵌入。
- `visual_slot_conflict`：如果同一段落已有原文证据截图，后续概念图必须跳过、改为替换或移动到下一小节；相邻图片和连续图片必须有文字承接和不同 claim，否则退回视觉计划。
- 封面必须在缩略图尺寸下可读，主体清晰，背景不喧宾夺主。
- 封面和正文插图必须通过本地图片质量检查：尺寸、比例、体积、非空白、亮度和对比度都合格。
- 封面默认由 Agent/image_gen 根据语义 brief 直接生成最终同名 `.png`；不要先生成 `{MMDD}_cover_bg.png` 再用 `gen_cover.py --background-image` 二次处理。
- 重新生成封面前必须更新封面 brief：补充文章主张、正文机制、品牌视觉元素、构图动线和避雷词；不得只在旧 prompt 上叠加“更高级、更牛逼”。
- `image_mode_awareness`：概念封面和插图必须先判断 `Mode B` / `Mode C`；没有宿主生图能力时只能输出 prompt 和状态说明，不得声称已经生成图片。
- `article_video_repurpose`：文章转视频必须先生成 `script.md` 和 `outline.md`，并确认节奏、素材和主题后再进入网页演示或媒体处理。
- 内置生图工具未暴露模型名时，视觉报告只能写“内置 image_gen / Agent 生成”；不得写成已确认的具体模型。
- 教程关键步骤必须有真实截图或可验证输出，不允许文字模拟终端截图。
- 插图必须支撑附近段落，不能只是装饰；生成 prompt 必须回应该段落的具体机制、场景、关系或数据，不得只按文章标题生成通用科技背景图。
- 生成图不得伪装成官方截图、新闻现场、产品 UI 或真实照片。

## 安全与合规

- 不输出 API Key、Cookie、token、账号、手机号、邮箱、身份证、地址、内部链接、登录态截图、私信、通知或带身份信息的侧栏。
- 不在正文写本地绝对路径，例如 `E:\...`；读者拿不到的资源应改成“后台回复获取”等公开路径表达。
- 医疗、法律、金融、政策、安全漏洞等高风险内容必须降低断言强度，标明来源和限制。
- 不编造采访、亲测、实验结果、用户反馈。
- 引用数据必须写明机构和时间；不写“多个来源显示”这种空泛来源。

## 审阅与发布

发布前必须完成完整审阅：

- 标题：长度、钩子、弱词、品牌准确性、AI 味。
- 事实：来源可靠、时效正确、无社区传闻冒充事实。
- 内容：信息密度、结构推进、读者收益、结尾自然。
- 价值立场：事实驱动发布稿以客观报道、正面意义和用户/消费者价值为主；边界说明中性、不过度、不抬杠。
- 人味：语言、结构、视觉都没有明显 AI 模板感。
- 重点句：突出显示只用于关键数据、机制转折、适用边界和行动建议；没有整段高亮、过度装饰或每段都强调的问题。
- 生产痕迹：正文、图注、alt、卡片、参考资料和结尾没有提示词、审阅说明、工作流规则或图片处理备注泄露。
- 排版：平台格式正确，移动端可读，章节层级清楚。
- 视觉：每张图有用途，事实/概念边界清楚。
- 原文证据截图：来源、翻译状态、裁剪范围、视觉槽位和支持 claim 清楚；没有登录态、私信、侧栏或相邻图片堆叠。
- 封面：能看出文章主张和品牌/产品语境；不是普通素材站科技图；用户反馈审美失败时必须重新生成封面。
- 发布：命令返回草稿 ID 或文章 ID；否则不算完成。

## 失败处理与可观测性

- 每个失败都要记录：失败阶段、原因、影响范围、下一步、是否可降级。
- P0 失败不得静默跳过。
