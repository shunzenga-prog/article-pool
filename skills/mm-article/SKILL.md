# mm-article

Article Pool 的语义优先多模态文章工作流。

## 首次加载

规划任务前，先读取 `workflow/mm-article/manifest.json`。把它当作语义任务、产物、质量门禁、宿主能力和旧流程隔离边界的工作流契约。

当任务进入真实创作、审阅或发布阶段时，必须读取 `references/production-standards.md`，并按其中的 P0/P1/P2 标准执行。该文件是生产级质量门禁，不是建议清单。

当任务涉及文章文件、配图、封面、截图或运行报告时，必须读取 `references/output-contract.md`。所有产物都要按该文件落盘。

当任务涉及本地资料、历史文章、知识库检索、模式感知生图、网页视频演示或视觉设计方向时，读取 `references/garden-creation-patterns.md`。它把 `ConardLi/garden-skills` 的方法改造成 Article Pool 自包含规则，不把外部仓库变成运行依赖。

## 核心规则

不要把这个工作流降级成一串脚本调用。模型负责语义规划，并决定需要哪些证据、文本、截图、生成图片、视觉检查和发布门禁。脚本只作为叶子节点上的确定性工具使用。

## 隔离规则

旧工作流继续有效，并且保持独立：

- 除非用户明确要求修改旧行为，否则不要修改 `skills/article-pipeline`、`skills/wechat-writer`、`skills/cover-gen` 或 `skills/illustration-gen`。
- 不要改变 `scripts/review_html.py`、`scripts/gen_cover.py`、`scripts/illustration_gen.py` 或 `scripts/publish_html.py` 的默认 CLI 行为。
- 如果用户要求使用旧工作流，直接使用旧 skill。
- 如果当前宿主没有多模态能力，优雅降级到兼容旧流程的确定性工具。

## 工作流

1. 识别平台、文章类型、时效性、目标字数和发布要求。
2. 根据 manifest 构建语义任务计划，不要照搬旧 pipeline stage。
3. 先做搜索来源规划和时效性校验，再收集证据，并把证据保存为带来源、时间和用途说明的产物。
4. 对会直接进入正文引用的原文页面、社交平台发帖、官方公告或论文页面，先执行 `evidence.source_capture`，产出 `source_capture_artifacts`，再进入标题、深度研究和起草。
5. 如果任务涉及本地资料、历史文章或知识库，调用 `article-research-kb`，产出 `local_kb_evidence_artifacts`，并把它和外部证据分开记录。
6. 生成标题并通过标题硬约束后，构造文章内容提示词。
7. 起草文章，并在文中标记视觉需求：截图、图表、封面、事实图片、生成插图。
8. 执行去 AI 味改写，再做平台排版和格式校验。
9. 对事实型视觉内容，优先使用真实截图、官方图片或浏览器/终端捕获结果。
10. 对概念型视觉内容，先判断 `Mode B` / `Mode C`：宿主有生图能力时走 Host-Native prompt 到本地图；没有生图能力时只输出 prompt 和状态，`不要假装出图成功`。
11. 如果用户要求文章转视频或网页演示，先生成 `script.md` 和 `outline.md`，再交给 `mm-video` 或网页演示实现。
12. 发布前完成审阅：标题、事实、时效、风格、排版、格式、视觉和发布条件。
13. 仅在边界明确的动作中使用确定性工具：HTML 审阅、封面后处理、图片嵌入、选题入库和草稿发布。

## 创作硬约束

### 标题生成约束

- 公众号标题必须 20-30 字；不足 20 字直接重写。
- 至少包含 2 个钩子元素：数字、反常识、紧迫感、承诺价值、制造好奇、身份认同、结果对比、具体对象。
- 禁止弱词：浅谈、浅析、关于、漫谈、初探、之我见。
- 禁止 AI 味标题句式，例如“从 X 看 Y 的 Z”“关于 X 的若干思考”“X 的机遇与挑战”。
- 最终标题必须写入 `title_artifact`，并说明字数、钩子组合和选择理由。

### 文章内容提示词

起草前先形成内容提示词，至少包含：

- 读者画像：谁会读、为什么现在要读、读完能得到什么。
- 研究问题：这篇文章真正要回答的 1-3 个问题；不能只是“某产品发布了什么”。
- 文章角度：主张、核心问题、不写什么，以及为什么这个角度比普通新闻复述更值得写。新闻发布、模型发布、产品发布类文章默认从客观事实、正面意义和用户/消费者价值出发，不要为了显得深刻强行唱反调。
- 事实边界：哪些内容必须引用来源，哪些是经验判断，哪些是假设。
- 指标拆解：如果涉及模型、价格、速度、榜单、性能、政策、融资等信息，必须说明关键指标衡量什么、不衡量什么。
- 机制解释：不能只列事实；必须解释导致结果的机制、链路、约束或系统结构。
- 限制边界：至少写出一个限制、反例、未验证点或替代解释，但只能做中性事实边界说明，不能写成抬杠、泼冷水或莫名反思。
- 读者决策：读者看完能如何判断、验证、迁移或行动；必须说明它可能帮用户/消费者解决什么问题，不能只得到“知道了”。
- 正向价值策略：事实驱动文章优先说明这件事带来的便利、效率、体验或选择空间；肯定必须有证据支撑，边界说明放在价值之后，不喧宾夺主。
- 开头策略：按文章类型选择事实导语、具体场景、冲突或困境。新闻发布、模型发布、政策公告等事实驱动文章，优先用事实导语交代“谁发布了什么、关键参数、开放范围、信息来源”，不要为了制造代入感硬编读者体验；不得一上来教育读者“应该怎么看”。
- 段落推进：每一节必须有自然转折和推进动作，不能只是提纲项的展开。
- 结构节奏：开头钩子、主体段落、转折、案例、结尾行动。
- 重点句突出计划：每个大节最多选择 1 句真正支撑判断、边界或行动的重点句，用微信兼容的 `strong + span` 或局部 `table` 做轻量突出；不要把普通形容词、空泛金句或整段文字高亮。
- 原文证据截图：如果已产出 `source_capture_artifacts`，提示词必须说明每张截图支持的 claim、建议插入位置和视觉槽位，避免同一位置连续插入多张图。
- 视觉计划：哪些段落需要真实截图，哪些可以用概念图，封面表达什么。
- 封面 brief：从正文主张提炼视觉主体、品牌视觉元素、构图和避雷词；不能只写“高端科技感封面”。
- 平台规则：公众号写 HTML，CSDN 写 Markdown，小红书写笔记，不允许机械互转。

推荐提示词骨架：

```text
请围绕「{选题}」创作一篇面向「{读者}」的「{平台}/{文章类型}」文章。
核心角度是：{角度}。
必须使用以下证据：{证据清单，含来源和发布时间}。
已采集的原文证据截图：{source_capture_artifacts，含来源 URL、截图路径、是否翻译、支持 claim 和视觉槽位}。
写作倾向：以客观事实和正面意义为主，从消费者/用户能解决什么问题出发；限制和边界只做事实说明，不写抬杠式反思。
标题约束：20-30 字，至少 2 个钩子，不用弱词，不要 AI 味句式。
正文要求：自然叙述，有人味，有具体判断，不编造亲历；事实和假设分清；必须包含研究问题、机制解释、正向价值、限制边界和读者可执行判断。
重点句要求：每个大节最多突出 1 句，优先突出结论、关键数据、适用边界或读者行动；突出样式必须克制、微信兼容，不做整段高亮。
视觉要求：标记每个截图、封面、插图的位置和用途；事实型视觉必须真实捕获；封面必须先有 cover_brief_artifact，概念图不得伪装事实，不得把未暴露底层模型名的内置生图说成具体模型。
输出格式：{公众号 HTML / CSDN Markdown / 小红书笔记}。
```

### 深度研究约束

对模型发布、产品发布、技术趋势、政策、融资、价格、榜单、事故、热点复盘等主题，起草前必须先形成 `research_depth_artifact`，至少包含：

- `root_question`：这篇文章真正要解答的问题，不得等同于选题标题。
- `shallow_angles_rejected`：至少列出 2 个被拒绝的浅层角度，例如“发布了什么”“国产模型崛起”“速度更快所以更好”。拒绝浅层角度不等于强行找负面角度；证据支持时，正向用户价值就是合格角度。
- `metric_map`：关键指标的定义、适用边界、不能推出什么。
- `mechanism_chain`：事实背后的因果链、工程链路、商业链路或政策链路。
- `comparison_frame`：至少一个同类产品、行业趋势、历史版本或技术路径参照。
- `counterpoints`：限制、反例、未验证点、替代解释或落地风险，只做中性边界说明，不写成抬杠式点评。
- `reader_payoff`：读者看完后能做出的判断、验证清单、行动选择，以及这项变化可能帮用户/消费者解决的具体问题。

如果文章只是把来源信息按顺序复述，或只在结尾加几句观点，判定为深度不足，不能进入排版。

### 写作手感约束

深度文章要先像人写，再像系统检查。起草前必须形成 `writing_craft_artifact`，至少包含：

- `reader_scene`：读者熟悉的真实使用场景、犹豫、阻塞或误解。
- `opening_strategy`：开头采用事实导语还是场景导语，如何自然转入主题，第一段不写什么；事实驱动文章不得强行场景化。
- `positive_value_plan`：哪些价值可以直接肯定，分别对应哪些证据、场景和用户收益；不得用空泛夸赞替代事实。
- `nut_graf`：用一小段说明这篇文章到底要回答什么，但不能写成报告摘要。
- `section_motion`：每一节往前推进什么，不得只是“事实/机制/影响/总结”的机械目录。
- `emphasis_plan`：哪些重点句需要突出、为什么突出、使用什么轻量样式；默认每个大节不超过 1 句，全文不超过 5-7 处。
- `transition_plan`：关键转场句，避免用“这意味着”“值得注意的是”“总的来说”等模板词。
- `stiffness_scan`：检查是否出现说教、强行引导、宏大空话、过度抽象、连续排比、抬杠式唱反调、为了深度而莫名反思，或用“我的判断/我的观点/我认为/我对这次发布的整体评价”把事实分析包装成个人表态。

禁止以下开头：

- “看到 X，最容易想到的是……”
- “如果一个模型……”
- “今天我们聊一个……”
- “最近，某公司发布了……”
- “在 AI 时代……”
- 直接抛抽象判断、宏大趋势或平台宣传语。

禁止以下观点包装：

- “我的判断：……”
- “我的观点是……”
- “我认为……”
- “我对这次发布的整体评价是……”
- “个人觉得/个人判断……”

发布、测评、模型对比类文章要让证据和结构自己得出结论。可以写“把这些变化放在一起看”“对使用者更直接的影响是”“更适合放在……任务里”，不要强调这是作者个人对事实的观点。

如果开头让读者感觉被引导、被教育、被作者预设反应，或新闻事实还没交代就硬编体验，必须重写；不能靠后文补救。

### 封面与生图质量约束

封面不是“科技感背景图”。生成封面前必须形成 `cover_brief_artifact`，并从文章内容提炼，而不是只复述标题。

`cover_brief_artifact` 至少包含：

- `article_claim`：这篇文章最想让读者记住的判断。
- `content_symbols`：正文里的 2-4 个核心机制、场景或物件，例如“时间缩微、逻辑折叠、系统互联、麒麟芯片”。
- `brand_visual_elements`：允许使用的品牌视觉元素。只能用抽象配色、构图、行业符号、产品类别或品牌语气；不得伪造 Logo、官方 UI、发布会现场或新闻图片。
- `composition`：主体、层次、动线、留白、缩略图焦点。
- `avoid_generic`：明确排除普通素材站科技图、泛化电路板壁纸、无意义光线、渐变球、bokeh、廉价 3D 图标。
- `model_attribution`：如果使用内置生图工具且工具没有暴露底层模型名，不得声称底层模型名；只能记录为“内置 image_gen / Agent 生成”。只有显式 CLI/API 返回模型配置时，才能写成具体模型。

封面 prompt 必须把 `article_claim`、`content_symbols` 和 `brand_visual_elements` 写进去。用户反馈封面“难看”“不像你的水平”“缺少某品牌元素”时，不要只加“更高级、更牛逼”这类空词；必须回到 `cover_brief_artifact`，补充正文机制、品牌元素、构图冲突和明确避雷词后重新生成。

如果封面看起来像普通素材站科技图，或品牌视觉元素只剩通用芯片、通用光线、通用红色，判定为审美失败，必须重新生成封面。

### 时效性校验

- 创作前必须确认资料发布时间，不能只看搜索结果摘要。
- 24 小时内：实时热点；24-48 小时：近期热点；超过 48 小时：转为分析、复盘或方法论角度。
- 新闻、产品发布、模型能力、价格、政策、榜单、融资、事故类信息必须重新搜索确认。
- 如果来源时间互相矛盾，优先官方公告、原始论文、项目 release、公司博客或监管文件。
- `timeliness_artifact` 必须记录发布时间、检索时间、时效分类和是否适合继续写。

### 搜索来源

搜索来源按优先级使用：

1. 官方/一手来源：官网、官方博客、release note、论文、GitHub 仓库、监管公告、公司公告。
2. 权威媒体/数据库：主流科技媒体、行业报告、学术数据库、可信新闻源。
3. 社区信号：Hacker News、Reddit、知乎、微博、X、论坛，只能用于判断热度或争议，不能单独支撑事实结论。
4. 图片和视觉来源：官方截图、产品页面、真实网页截图、GitHub/文档截图优先；概念图只能表达抽象概念。

每个关键事实至少有 1 个可靠来源；争议性或高风险判断尽量交叉验证 2 个以上来源。

### 原文证据截图

原文证据截图是证据层产物，不是普通配图。凡是正文要引用“某人/某机构在网页、X、论坛、官方页面上说了什么”，优先在起草前执行 `evidence.source_capture`，产出 `source_capture_artifacts`。

- 截图优先使用真实网页版；如果读者需要中文理解，可以使用浏览器翻译后的页面，但报告里必须记录 `translated=true` 和原始 URL。
- 截图范围只保留头像/账号/正文/必要时间信息；不要截浏览器导航、侧栏、视频大块区域、评论噪声或无关推荐。
- 对 X 等社交平台发帖，默认使用紧凑裁剪：头像、显示名、handle、正文文字、必要时间信息；需要展示原貌时才保留更大页面截图。
- 每条 `source_capture_artifacts` 必须记录 `source_url`、`captured_at`、`screenshot_path`、`claim_supported`、`translated`、`crop_style`、`visual_slot_id` 和 `reader_use`。
- 截图不能暴露登录态、Cookie、私信、通知、内部链接、用户隐私或本地绝对路径；发现这些内容时必须重新裁剪或放弃截图。
- 正文图注只能写读者需要的信息，例如“图：Google Magenta 在 X 上介绍 MRT2 的 MIDI 控制能力”，不能写“裁切版”“视觉计划”“截图处理”等生产备注。

### 视觉槽位协调

`source_capture_artifacts` 会先占用视觉槽位。后续 `plan.visuals`、`capture.factual` 和 `generate.image` 必须读取这些槽位，避免同一位置连续插入多张图。

- 每个视觉需求都要有 `visual_slot_id`、段落位置、用途和事实/概念属性。
- 如果某段已经放了原文证据截图，默认跳过该段的概念插图；确有必要时只能改为替换，或移动到下一小节中有独立叙事作用的位置。
- 同一位置不得连续插入多张图。两张图片必须有明确文字承接、不同 claim，且不造成“截图后又堆一张泛科技图”的阅读噪声。
- 教程截图、产品 UI 截图和原文证据截图都属于已占用槽位；概念图只能补充抽象机制，不能贴在同一段落制造图像堆叠。
- 视觉审阅时要检查相邻图片、连续图片、同一段落重复配图；发现冲突时回到视觉计划重排。

### 排版

- 公众号正文直接从导语开始，不重复系统标题。
- 公众号使用微信安全 HTML：文字样式放在 `span`，段落只保留必要 inline margin 和 `text-align`。
- 禁止用 `div`、`section` 做容器；`table` 只做局部卡片，不包裹全文。
- 章节标题必须有 `border-bottom` 下划线，不能只靠加粗和放大。
- 重点句突出优先用 `<strong><span style="background-color:#fff3d8;color:#23302b;">...</span></strong>` 这类微信兼容内联样式，或用局部 `table` 做信息卡；不要在 `<p>` 上放字体、颜色或背景样式。
- 全文只使用一个主色系和最多一个点缀色，避免大面积高亮和模板感。
- 教程类文章每个关键步骤要有真实截图或可验证输出，不能用文字模拟终端截图。

### 格式校验

- 公众号：输出 `.html`，发布前运行 `python scripts/review_html.py <文章.html> --json`；教程类加 `--tutorial`。
- CSDN：输出 `.md`，检查 `#` 标题、`##` 章节、引用、表格、代码围栏和标签。
- 小红书：输出笔记结构，标题、正文、封面建议、标签分开。
- 文章文件名、封面、插图版 HTML 要遵守 repo 约定；不要把本地绝对路径写进正文。
- 格式不通过时，回到对应语义任务修复，不允许直接进入发布。

### 去 AI 味

- 避免套话：值得注意的是、总的来说、综上所述、在当今时代、不可否认的是。
- 避免抬杠腔：不要为了显得深刻而写“但是/然而/真正的问题是/这不是重点”等强行转折；优先用顺承式表达说明事实、价值、适用范围。
- 避免报告腔：不要机械编号，不要每段都用同样句式，不要“背景-问题-解决方案-总结”模板化到底。
- 避免第一人称观点腔：发布、测评、模型对比类文章不要用“我的判断”“我的观点”“我认为”“我对这次发布的整体评价”做章节标题或结论引导；改用事实归纳、使用场景和读者决策来收束。
- 增加真人感：具体判断、有限主观感受、真实限制、犹豫点、反例、读者熟悉的细节。
- 重点句突出要服务阅读节奏：只标出关键数据、选择边界、机制转折和行动建议；不把每段第一句、普通形容词、口号式金句都高亮。
- 保持诚实：没有亲测就不要写“我试过”；没有采访就不要写“我问了”；假设性内容必须标明。
- 视觉也要去 AI 味：少用整段高亮、渐变堆叠、过度卡片化和无意义装饰。
- 正文、图注、alt、卡片和结尾不得出现工作流提示词或制作者备注，例如“事实型图片优先使用官方截图”“不用生成图替代”“裁切版”“视觉计划”“图片请求”“审阅通过”等面向创作者而非读者的语句。

### 审阅

发布前必须做完整审阅，至少覆盖：

- 标题：20-30 字、至少 2 个钩子、无弱词、无 AI 味。
- 事实：关键事实有来源，时效性分类正确，没有把社区传闻当事实。
- 内容：开头有钩子，主体有信息密度，结尾有自然行动，不空泛。
- 价值立场：发布类文章以客观事实和正面意义为主，能从消费者/用户解决什么问题出发；限制边界中性说明，不抬杠、不莫名反思。
- 去 AI 味：语言、结构、视觉都要检查。
- 读者可见泄露：图注、正文、alt 和卡片里不能出现提示词、审阅说明、工作流规则、文件处理说明或“裁切版”等生产备注。
- 排版：主色统一、章节标题有下划线、卡片不过量、移动端可读。
- 格式：公众号 HTML/CSDN Markdown/小红书笔记各走自己的格式，不互相机械转换。
- 视觉：每张图都对应段落；生成图不伪装事实；封面缩略图可读。
- 发布：只有审阅通过，且目标平台返回草稿 ID 或文章 ID，才算完成。

## 产物契约

每个有意义的输出都要作为产物跟踪：

| kind | 示例 | 必填字段 |
| --- | --- | --- |
| `evidence` | 搜索结果、官方文档、文章来源 | `path_or_url`, `captured_at`, `claim_supported` |
| `source_capture` | 原文证据截图、翻译后网页紧凑截图 | `source_url`, `screenshot_path`, `claim_supported`, `visual_slot_id` |
| `draft` | HTML 或 Markdown 文章 | `path`, `platform`, `title` |
| `visual_plan` | 封面、插图、截图清单 | `items`, `rationale` |
| `cover_brief` | 封面视觉 brief | `article_claim`, `content_symbols`, `brand_visual_elements`, `composition`, `avoid_generic`, `model_attribution` |
| `image` | 生成封面、生成插图、已上传图片 | `path`, `source`, `intended_use` |
| `screenshot` | 浏览器、终端、界面、代码输出截图 | `path`, `source_command_or_url` |
| `review` | HTML 审阅、视觉审阅、发布审阅 | `passed`, `issues`, `next_action` |
| `publish_result` | 微信草稿或 CSDN 发布结果 | `platform`, `id`, `url_or_status` |

## 输出规范

详细目录、命名、配图流程和封面流程见 `references/output-contract.md`。核心约定：

- 公众号文章默认参考老文章池命名，但根目录固定为 `/Users/mulin/workspace/公众号/文章`，输出到 `/Users/mulin/workspace/公众号/文章/{YYYY}年{MM}月/{MMDD}-{safe_title}.html`。
- 插图版输出为同名 `_illustrated.html`，发布脚本生成的 CDN 版本为同名 `_cdn.html`。
- 封面输出为同名 `.png`；当前 Agent 具备 image_gen 能力时，直接生成最终封面到该路径，不再生成 `{MMDD}_cover_bg.png` 再交给脚本二次处理。
- 本地截图和 Agent 生成图与 HTML 或 Markdown 文件平铺在同一个月份目录下，不创建正式文章图片子目录；原文证据截图使用 `{MMDD}-{safe_title}-source-{platform}-{NN}-compact.png`。
- 运行记录放入 `reports/mm-article/{run_id}/`，包括证据、原文证据截图、标题决策、内容提示词、视觉计划、图片请求、生成图片清单、审阅和发布结果。
- 封面生成必须记录 `cover_brief_artifact` 或等价字段；如果用户反馈封面，需要记录反馈点和下一版 prompt 的变化。
- 正文里不得出现本地绝对路径；本地图必须在发布前上传或替换为可访问 URL。

## 宿主能力决策

- 如果 Codex 生图能力可用，概念型封面和概念型插图优先生成本地图片。
- 使用内置生图工具时，如果宿主没有暴露底层模型名，不得声称底层模型名；模型归因必须按实际可见工具能力记录。
- 如果浏览器或截图工具可用，事实型界面、教程证据和原文证据截图必须优先真实捕获，不要凭空生成。
- 如果只有文本工具可用，保留语义计划，但使用旧流程安全兜底，并标记视觉置信度不足。
- 绝不要用生成图片伪造截图、产品界面、官方 Logo 或现实证据。

## 确定性工具边界

只有当输入已经由语义层决定后，才使用这些工具：

- `python scripts/review_html.py <article.html>`：用于微信 HTML 兼容性门禁。
- `python scripts/gen_cover.py --title "<title>" --output <cover.png>`：legacy 兜底，仅在无 image_gen 能力或用户明确要求真实图库/事实图片时使用；默认封面不得调用该脚本。
- `python scripts/illustration_gen.py <article.html> --emit-image-requests <requests.json> --dry-run`：用于生成概念图片需求清单；每条请求必须包含 `paragraph_context`，生图 prompt 要根据插图所在段落的具体内容生成，不能只按标题词或主题词泛化。
- `python scripts/illustration_gen.py <article.html> --use-local-images <generated_images.json>`：用于嵌入已经生成的本地图片。
- `python scripts/validate_mm_delivery.py <article.html> --run-dir reports/mm-article/<run_id> --title "<title>" --write-report`：用于最终交付门禁，检查封面、插图、图片质量、HTML 审阅、流程泄漏、运行报告和发布状态。
- `PYTHONIOENCODING=utf-8 python scripts/publish_html.py <article.html> --cover <cover.png>`：在 Windows 上发布到微信公众号草稿箱。

## 质量门禁

宣布任务完成前，必须满足：

- 对时效性或事实性内容，文章中的说法必须有来源支撑。
- 每个视觉产物都要对应它支持的段落或封面目标。
- 原文证据截图必须先作为 `source_capture_artifacts` 进入证据台账，并和后续配图共享视觉槽位。
- 同一段落或同一视觉槽位不得连续插入多张图；后续配图要跳过、替换或移动。
- 正文插图生成时必须先读 `image_requests.json` 里的 `paragraph_context`；画面要回应该段落的机制、场景或关系，而不是做通用科技背景图。
- 封面必须能回到 `cover_brief_artifact`，并体现文章主张、内容符号和合法品牌视觉元素。
- 概念图必须明确是概念图，不能伪装成截图或新闻照片。
- 发布到微信前，HTML 必须通过微信兼容性审阅。
- 封面和首屏视觉在缩略图尺寸下仍然可读。
- 宣布文章交付前，必须运行 `validate_mm_delivery.py` 并把结果写入 `delivery_gate.json`。
- 除非用户明确要求 dry run，否则只有目标平台返回草稿 ID 或文章 ID，才算发布完成。
