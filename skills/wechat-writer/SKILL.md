# 公众号文章写作与排版

> **定位：** article-pipeline 的 Stage 2「创作官」。只负责写内容和排版，不负责审阅、插图、封面、发布。
> **入口：** 由 article-pipeline 调用。如果被单独触发，完成写作+排版后必须提醒调用方走 pipeline 后续阶段。

## 触发
- 由 `article-pipeline` Stage 2 调用，不单独使用

## ⚠️ 核心原则
### 爆款公式
```
爆款 = 热点选题 × 情绪价值 × 结构化表达 × 视觉呈现
```

### ⚠️ 禁止暴露 AI 身份（核心！）

| ❌ 禁止 | ✅ 正确 |
|--------|--------|
| "作为 AI 助手" | 直接表达观点 |
| "我是 AI" | 无需说明身份 |
| "作为一个人工智能" | 直接写观点 |
| "小咪看到这个消息心情复杂" | "这个消息让人心情复杂" |

**小咪的定位：**
- ✅ 可以用第三人称"小咪"（笔名/昵称）
- ✅ 可以有观点、情绪、态度
- ❌ 不能说"我是 AI""作为 AI 助手"

### ⚠️ 去 AI 味写作（核心！）

| ❌ AI 味太重 | ✅ 自然写作 |
|------------|------------|
| 未来展望分"短期/中期/长期" | 用几句话自然收尾 |
| "小咪观点"搞大标题 | 结尾自然表达想法 |
| 引用原文用翻译后的语言 | 原话说出来 |
| 大量颜色高亮语法 | 适度强调，简洁为主 |
| 结构化分层（一、二、三） | 流畅叙述，自然过渡 |
| 完全无样式，太平淡 | 适当加粗、引用，有层次感 |

**样式使用原则：**
- ✅ 重点内容可以**加粗**
- ✅ 引用原文用引用格式（灰色边框）
- ✅ 关键数据可以突出
- ❌ 不要整段高亮颜色
- ❌ 不要过度使用<span style="color:red">

**核心：有层次感，但不像写报告！**

### ⚠️ 时效性验证（必查！）
**创作前必须验证每条信息的时效性：**
- 搜索确认事件发生时间
- 检查是否有最新进展/辟谣
- 标注超过30天的信息为"旧闻"
- 避免使用过时信息当新闻

**过时信息判断原则：**
- 搜索时确认模型/产品的最新版本号，避免引用已被替代的旧版本
- 人物/公司动态类信息需确认是否为近期事件
- 如有疑问，用 web_search 搜索 "xxx 最新" 确认时效性

**时效规则：**
- <24h：🔥 实时热点
- 24-48h：📈 近期热点
- >48h：💡 转分析角度，不当新闻

### ⚠️ 选题查重（创作前必做）

**目的：** 防止同一话题在保护期内重复创作，保持内容多样性。

**操作步骤：**
1. 先读取 `reports/used_topics.json` 了解近期已用选题
2. 将你打算写的选题核心关键词与已有记录对比
3. 做**语义判断**（不是精确匹配）：

**判断示例：**
| 新选题 | 已有记录 | 判断 |
|--------|---------|------|
| "GPT-5 正式发布" | "OpenAI 增速预警" | ✅ 不重复 — 不同角度 |
| "OpenAI 营收不及预期" | "OpenAI 增速预警" | ❌ 重复 — 同一事件 |
| "谷歌 AI 芯片突破" | "OpenAI 增速预警" | ✅ 不重复 — 不同公司 |
| "AI 概念股集体跳水" | "OpenAI 增速预警" | ⚠️ 可能重叠 — 需注意角度区分 |

**关键词选取参考：**
- 提取 3-5 个核心词：公司名、产品名、事件主题、行业标签
- 覆盖文章涉及的 2-3 条主要话题线

**如果查重不通过：**
- 换角度（技术→商业，国内→国际）
- 换话题（选其他热点）
- 等保护期过后再写

**辅助命令：**
```bash
python scripts/topic_tracker.py list          # 查看保护期内选题
python scripts/topic_tracker.py list --days 14  # 自定义天数
```

### 三大禁忌
1. ❌ **表格太多** - ≤2 个/篇
2. ❌ **论述太浅** - 需深入分析
3. ❌ **缺少真实感** - 禁止编造"我试过了"

### ⚠️ 表情符号
- **标题**：❌ 不用 | **正文**：⚠️ 极少 | **结尾**：✅ 1 个（🐱）

### ⚠️ 人味写作
- ❌ "该工具具有以下特点" → ✅ "根据文档和社区反馈，这工具有几个亮点"
- **禁止编造**：不能写"我试过了"、"我亲自体验过"

### 小咪特色
- 🐱 身份标识 | ✅ 真实来源 | 💡 金句（≥1 句） | 🎯 行动号召

## 创作流程

> **这不是线性流水线——后面步骤发现问题时，回到对应步骤修改，直到通过。**
> **本流程在步骤 11 结束。排版完成的 HTML 交还 article-pipeline，由 pipeline 继续审阅→插图→封面→发布。**

1. **选题** → 热点/教程/观点
2. **查重** → ⚠️ 读取 `reports/used_topics.json`，确认选题不与近期内容重复（详见「选题查重」）
3. **时效** → <24h 可用，>48h 转分析
4. **风格规划** → ⭐ 定一张「风格卡」（6维度：基调+色系+背景层次+功能区域+强调+节奏），后续所有写作和排版都以此为指南（详见下方「风格规划」）
   - ⚠️ 定完风格卡后，必须在本地存档中找 1 篇视觉效果好的文章作为"视觉基准"，扫一眼确认风格卡至少覆盖了基准文章的视觉层次丰富度
5. **标题** → 20-30 字，爆款公式。带着风格卡的基调写——基调决定标题的语气
6. **开头** → 3 秒钩子（见 `references/viral-guide.md`）
7. **正文** → SCQA，每段 3-5 行。写的时候就用风格卡的强调方式——数据用辅色加粗、关键句用强调色
8. **结尾** → 行动号召 + 金句
9. **去 AI 味（文字层）** → ⭐ 检查结构化分层、AI 腔、编造感。不通过 → 回步骤 7
10. **即时排版** → 根据风格卡即时生成 HTML（详见下方「即时排版」）
11. **去 AI 味（视觉层）** → 检查整段高亮、过度用色、过度装饰。不通过 → 回步骤 10

**迭代回路速查：**

```
步骤 9 不通过  → 回步骤 7（正文）
步骤 11 不通过 → 回步骤 10（排版）
```

## 交还 Pipeline

**步骤 11 通过后，产出以下内容，交还 article-pipeline 继续：**

| 产出 | 说明 | 下游消费者 |
|------|------|-----------|
| HTML 文件 | `文章/{年份}年{月份}月/{日期}-{标题}.html` | Stage 4 审阅官 |
| 风格卡 | 已执行的配色/强调/节奏方案 | Stage 4 视觉检查 |
| 标题 | 20-30 字最终标题 | Stage 4.8 封面生成 |

**article-pipeline 接管后的流程：**

```
Stage 4 审阅官   → review_html.py 硬检查 + 软检查
Stage 4.5 插图   → illustration_gen.py 自动配图，输出 _illustrated.html
Stage 4.8 封面   → gen_cover.py auto 模式，>100KB
Stage 8 发布官   → publish_html.py 推送到草稿箱 + 题目入库
```

本技能不再处理审阅、插图、封面、发布。这些由 article-pipeline 的对应 Stage 负责。

## 标题钩子公式（8 个）⚠️ 核心

**铁律**：标题决定 80% 打开率。必须用钩子，不能用陈述句。

| # | 公式 | 结构模板 | 案例 |
|---|------|---------|------|
| 1 | **反常识** | [常识] + 但/却 + [反转] | "AI 写的代码 bug 更少，但没人敢用" |
| 2 | **利益承诺** | [结果] + 只需/不用 + [低成本] | "搭一条自动流水线，每天多出 3 小时" |
| 3 | **恐惧损失** | 不/别 + [行动] + 否则 + [后果] | "再不学 AI Agent，明年面试就没人要了" |
| 4 | **数字对比** | [数据A] vs [数据B]：[反差结论] | "7250 亿投入和 52000 个裁员：AI 的冰与火" |
| 5 | **悬念好奇** | [事件] + 背后/原因/一句话 | "黄仁勋演讲中，有一句话被所有人忽略了" |
| 6 | **身份认同** | [人群] + 都/正在 + [行为] | "真正会用 AI 的人，都在偷偷做这件事" |
| 7 | **场景痛点** | [场景] + 还在/每次 + [痛点] + ？ | "写周报还在憋 2 小时？一个 prompt 搞定" |
| 8 | **新概念** | [新词]： [一句话定义 + 价值] | "Vibe Coding：不写代码，用嘴把产品做出来" |

### ⚠️ 标题硬约束（不可跳过）

- **字数**：20-30 字（不足 20 字 = 驳回重写）
- **钩子元素**：每个标题 ≥2 个（数字 / 反常识 / 紧迫感 / 承诺价值 / 制造好奇 / 身份认同）
- **至少包含**：1 个具体数字 或 1 个情绪词
- **禁止弱词**：❌ "浅谈""浅析""关于""漫谈""初探""之我见"

### 标题自检清单（生成后必过）

```
[ ] 20-30 字？
[ ] ≥2 个钩子元素？
[ ] 有具体数字或情绪词？
[ ] 没有"浅谈/浅析/关于/漫谈/初探"？
[ ] 读一遍——你想点吗？
```

**更多案例**：见 `references/viral-guide.md`

## 🎨 风格规划（创作前必做）

**每篇文章的配色由 AI 根据选题气质自主生成，不锁死具体色值。** 模板（`templates/`）作为兜底保留。

### 设计原则（必须遵守）

1. **一个主色系贯穿全文**——底色、卡片、分隔线、标签都从同一色系衍生，仅靠深浅变化产生层次。严禁暖色区和冷色区混搭。
2. **点缀色最多 1 个**——仅用于关键数字和章节标题下划线，用量 <5%。
3. **章节标题必须有下划线**（`border-bottom`）——不能只靠加粗+放大。
4. **章节之间必须可见分隔**——`···` 用于章节内小过渡，大章节间用更明显的分隔。
5. **卡片是调料不是主菜**——大部分文字放页面上，卡片只在少数锚点出现。
6. **内容在 `<p>` 中自然流动**——不要用 `<table>` 包裹全文（手机宽度问题和黑缝），`<table>` 仅用于数据卡片等局部区块。

### 风格卡格式

创作前用 3-5 句定一张风格卡，描述配色方向和节奏即可，不需要写具体色值：

```
风格卡：
- 主色方向：[暖调 / 冷调 / 中性？跟文章情绪的关系？]
- 点缀方向：[金色系 / 蓝色系 / 红色系？只用在全文字数 <5% 的地方]
- 节奏：[大部分文字自然流动，卡片在哪里收一下？]
```

### 风格卡示例

```
风格卡（深度分析：AI 开源项目）：
- 主色方向：暖灰调，沉稳但不冷，适合技术深度分析
- 点缀方向：琥珀金，只用于关键数据和章节标题下划线
- 节奏：开头摘要 + 性能数据 + 结尾金句各用一个卡片收住，其余文字自然流动
```

```
风格卡（技术教程：AI 数据分析）：
- 主色方向：冷灰调，干净理性，适合代码和图表
- 点缀方向：钢蓝，标注正确输出和关键步骤
- 节奏：每个步骤一个浅色卡片，代码块暗底，错误提示红标
```

### 模板的定位

模板从"主角"变为"兜底"：
- **首选**：根据风格卡即时生成 HTML（更贴合内容气质）
- **兜底**：当需要快速产出或风格卡与模板高度匹配时，用 `templates/` 下的模板（通用 `article-template.html`、教程 `tech-tutorial.html`、深度 `daily-report.html`）
- 即使用模板，也应根据风格卡调整颜色等细节，不要照搬

## 📐 即时排版（替代模板套用）

**根据风格卡即时生成 HTML，而不是填模板。**

### 核心约束（微信 CSS，不可商量）

| 约束 | 说明 |
|------|------|
| 容器 | 必须用 `<table><tr><td>`，禁止 `<div>`/`<section>` |
| 文字样式 | 必须放在 `<span style="...">` 上，不要在 `<p>` 上放 color/font-size |
| 章节标题 | 不能只靠加粗+放大。用 `border-bottom` 下划线（点缀色，2px），让标题有"锚点感" |
| 章节分隔 | 大章节之间必须交替底色（同色系深浅交替）或插装饰线。`···` 仅用于章节内的轻过渡 |
| 代码块 | 用 `<pre style="...">` 暗底浅字 |
| 图片 | 用 `<img width="100%">` 属性控制，外层 `<td>` 包裹 |
| 段落 | `<p>` 只放 `margin` + `text-align`，其余样式下沉到 `<span>` |

完整约束见 `references/wechat-css-guide.md`。

### 即时生成的要点

1. **颜色严格 ≤4 种**（风格卡定义的主色+辅色+强调+正文色）
2. **段落 ≤5 行**，超过自动拆分
3. **图文紧贴**，不给读者跳转的间隙
4. **代码块 3-20 行**，只展示关键代码
5. **引用框/思考框**用 `<td>` + `border-left` 实现，保持简约

### 参数默认值（风格卡未指定时使用）

| 元素 | 默认值 |
|------|--------|
| 正文字号 | 15px |
| 行距 | 1.8 |
| 标题字号 | 18px |
| 代码块 | 背景 #1e1e2e，字号 13px |
| 段落间距 | margin: 8px 0 |

## 去 AI 味（视觉层）

文字去完 AI 味后，排版还可能出现"视觉 AI 味"——这是容易被忽略的盲区。

| ❌ 视觉 AI 味 | ✅ 自然排版 |
|-------------|-----------|
| 整段文字用彩色高亮（红底/蓝底大段文字） | 仅关键数字或短句用颜色 |
| 冷暖色系混搭（蓝卡片 + 黄卡片 + 红标签）← **最常见** | **一个色系贯穿全文**，仅用深浅变化和 1 个点缀色 |
| 大段渐变背景、阴影、圆角（公众号会剥离） | 扁平、干净，用左边框或底部线做分隔 |
| 每个段落包一个卡片，文章像 UI 组件展示页 | 大部分文字自然流动，卡片只在关键锚点出现 |
| 代码块用亮色背景（白底代码块在手机端刺眼） | 代码块统一暗底浅字 |

**检查方法：** 生成 HTML 后，退一步看。能不能用一个形容词描述全文的色调？（"暖棕的""冷灰的""深蓝的"——一个词能概括就对了，需要两个词说明已经混了。）

## 📸 教程类文章截图规范（必读）

**教程文章的核心竞争力是"实操感"——读者要看到真实的执行过程和结果，不是文字模拟。**

### ⭐ 核心原则：分步截图，同步嵌入

**写一步 → 截图一步 → 嵌入一步 → 再写下一步。** 这是教程流水线与通用流程最本质的区别。

教程不是写完所有文字再回头补图。原因：
1. 代码执行结果可能变化（API 数据、时间戳），回头补图会和文字对不上
2. 写完再截图容易漏掉关键步骤
3. 同步截图能第一时间发现"代码跑不通""输出不一致"等问题

### 截图类型（按优先级）

| 优先级 | 类型 | 说明 | 生成工具 |
|--------|------|------|---------|
| 0 | 流程图/架构图 | 流水线全景、系统架构、数据流 | Mermaid + Playwright 渲染 |
| 1 | 终端执行截图 | 命令运行的真实输出，逼真终端窗口 | terminal_screenshot.py（xterm.js + Playwright） |
| 2 | 界面/浏览器截图 | 生成的页面、工具界面、最终效果 | Playwright 全页截图 |
| 3 | 原文/源数据截图 | 点击链接后的实际内容页，证明数据真实 | Playwright viewport 截图 |
| 4 | 效果对比截图 | Before/After，或不同方案的对比 | 组合以上工具 |

### ⚠️ 截图规划（写生成脚本前必做）

**最容易犯的错误：用终端渲染工具生成所有截图。** 必须提前规划每张截图用什么工具。

**操作步骤：**
1. 列出文章中需要的所有截图
2. 对照上面的类型表，给每张图打上类型标签
3. 根据类型选择对应的生成工具
4. 再写生成脚本

**示例（本期就是反面教材）：**

| 截图 | 实际类型 | 应该用的工具 | 错误做法 |
|------|---------|-------------|---------|
| 终端运行输出 | 终端截图(类型1) | terminal_screenshot.py | ✅ |
| HTML 报告 | 界面截图(类型2) | Playwright | ❌ 用终端渲染画 ASCII 框 |
| 图表 | matplotlib 输出 | 直接保存 chart | ✅ |

**原则：** 终端输出 → `terminal_screenshot.py`（xterm.js 渲染，自动匹配操作系统标题栏）。网页/报告 → Playwright 浏览器截图。各用各的工具，不要混用。

### 数量要求

- **每个教程步骤至少 1 张截图**，execute 和 output 类步骤必须有图
- **每 1500 字至少 1 张截图**（最低标准）
- 截图总数与步骤总数比例不低于 **0.8:1**
- **用实际运行的真实输出，禁止用文字代码块模拟终端效果**

### 截图生成命令

> **操作命令见** `skills/tutorial-pipeline/SKILL.md` Stage 2「截图生成工具速查」。
> 核心渲染引擎统一由 `scripts/capture/` 提供：`terminal_screenshot()` / `browser_screenshot()` / `code_to_image()` / `flowchart_to_image()`。

### 图片嵌入规范

- 使用 `<table><tr><td>` 包裹 `<img>`，确保发布后 style 不被剥离
- `<img width="100%">` 用属性而非 CSS 控制尺寸
- alt 文本写清楚截图内容（便于读者理解，也便于搜索）
- 每张截图前有引导文字说明"这是什么"

### ⚠️ 字体配置（防止乱码）

> **统一字体管理见** `scripts/capture/fonts.py` — `FontManager` 跨平台自动检测 CJK/等宽/拉丁字体，14 路径回退链。`scripts/capture/chart.py` 的 `ChartCapture` 自动注入字体配置。
>
> **用法：** `from capture.fonts import get_font_manager; fm = get_font_manager(); fm.cjk_path`
>
> matplotlib 图表使用 `chart_screenshot()` 会自动处理字体问题，无需手动配置。其他截图工具（terminal/browser/code/flowchart）各自通过 `BaseCapture.fonts` 获取字体。

### 禁止事项

- ❌ 用文字代码块模拟终端输出——读者一眼识破，丧失信任
- ❌ 截图里出现本地绝对路径（如 `E:\WorkSpace\...`）——用示例路径
- ❌ 文章里写"完整源码: 教程资源/xxx/xxx.py"这种本地路径——公众号读者看不到
- ❌ 只在文末说"代码在xx"，正文没有任何截图做证据

## 📝 教程类文章代码讲解规范（必读）

> **教程写作优先参考 `skills/tutorial-pipeline/SKILL.md` 「教程写作深度标准」。** 该标准覆盖了本文的简洁规则（段落 ≤5 行、表格 ≤2 个等不适用于教程）。本文的"代码讲解三要素"为通用参考。

**教程文章的核心价值不是"展示结果"，而是"解释过程"。** 读者看教程是为了学会自己怎么做，不是看 AI 怎么炫技。

### ⭐ 代码讲解三要素（适用于 code/execute 类步骤）

setup/config 类酌情适用，overview/output/summary 类不适用。

对每个关键函数或代码步骤：

1. **代码片段** — 实际的关键代码（3-20行），不是伪代码或函数名列表
2. **逻辑解释** — 逐行/逐段说明代码在做什么，为什么这样写
3. **设计思路** — 为什么要选这个方案？有什么替代方案？踩了什么坑？

### 正反对比

| ❌ 错误：只描述结果 | ✅ 正确：展示代码 + 解释思路 |
|--------------------|---------------------------|
| "AI 用 IQR 法检测了异常值" | 展示 IQR 的计算代码：`q1, q3 = df.quantile([0.25, 0.75])`<br>解释：Q1 是 25% 分位数，1.5 倍 IQR 是统计学通用标准<br>说明：只标记不删除——因为高销售额也可能是真实大单 |
| "AI 自动选择了 utf-8-sig 编码" | 展示：`pd.read_csv(csv, encoding="utf-8-sig")`<br>解释：Excel 导出的 CSV 带 BOM 头，utf-8 读到列名会多个鬼字符<br>对比：utf-8 vs utf-8-sig 的区别就是自动去掉 BOM |
| "AI 生成了 5 张图表" | 挑一张图表展示完整代码（15-30行）<br>解释：figure→axes→plot→annotate→save 的 matplotlib 标准流水线<br>说明：为什么用 barh 而不是 bar（横向更利于手机端查看） |
| 只列 5 个函数名 | 展示 `main()` 函数代码<br>解释：为什么设计成 5 步流水线？每个函数只做一件事，方便单独修改<br>说明：如果加新图表只需改 `make_charts()`，不影响其他步骤 |

### 代码展示频率（按步骤类型区分）

| 步骤类型 | 代码展示要求 |
|---------|------------|
| code / execute | 必须有代码片段，挑 2-3 个关键点做深度讲解 |
| setup / config | 酌情——`pip install` 之类不需要；涉及配置坑点可以贴代码 |
| overview / output / summary | 不需要代码展示 |

- 代码片段 3-20 行最佳（太短没意义，太长手机端溢出）
- 一篇文章有 2-3 个代码块到达"深层"即可，全篇深层反而让读者疲劳

### 代码讲解的层次

```
表层（展示）   → "AI 写了这段代码"
中层（解释）   → "这行代码做了什么"
深层（思考）   → "为什么这样做，而不是那样做"
```

**一篇文章有 2-3 处到达"深层"就足够好——读者需要的是关键洞见，不是逐行注释。**
